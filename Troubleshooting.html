6.1 Troubleshooting Build Issues

  • clang-mutate is not on my path
  • It takes a long time to compile an instrumented project
  • CURRY-COMPOSE-READER-MACROS::LCURLY-BRACE-READER is undefined
  • Deleting cache files

clang-mutate is not on my path

SEL assumes that the clang-mutate executable (see clang-mutate) is available on the shell’s $PATH.

  • Ensure that clang-mutate is on your $PATH.

If this is not sufficient, the path used by SBCL may not be inheriting $PATH changes made in your user environment. In this case:

  • place a clang-mutate executable on the standard system search path, or
  • update the path in /etc/profile.

It takes a long time to compile an instrumented project

Instrumentation may add thousands of lines of code to your project, leading to a dramatic increase in compilation time.

The following will often speed up compilation.

  • Use clang as your compiler.
  • Compile without optimizations enabled (-O0).

CURRY-COMPOSE-READER-MACROS::LCURLY-BRACE-READER is undefined

This usually indicates an issue with cache files created by ASDF.

To resolve this issue:

  • select the try-recompiling restart at the repl, or
  • delete cache files.

Deleting cache files

As part of the compilation process, ASDF saves compiled files to a cache as described in the ASDF manual. When dealing with some compilation issues (see CURRY-COMPOSE-READER-MACROS::LCURLY-BRACE-READER is undefined), it may be necessary to delete the cache.

By default, the cache should be located in ~/.cache/common-lisp/[lisp-implementation]/path/to/local/projects/software-evolution-library.

To remove the cache, execute: 

    rm -rf ~/.cache/common-lisp/[lisp-implementation]/path/to/local/projects/software-evolution-library

6.2 Troubleshooting Test Issues

  • Did all unit tests pass?
  • There are many test failures
  • Unit tests aren’t running on my 64-bit machine

Did all unit tests pass?

If the make check build target shows a sequence of dots with no ’E’ or ’X’ characters, all unit tests passed. See the stefil project page for more information on our unit testing framework.

There are many test failures

A large number of test failures usually indicates a major prerequisite is missing. To debug, try the following.

  1. Ensure clang and clang-mutate are on your $PATH.
  2. Execute 
    clang -m32 -g -O0 test/etc/hello-world/hello_world.c
  3. Execute 
    clang-mutate -json test/etc/hello-world/hello_world.c -- -g -O0 -m32

If any of the above fail, the unit test failure is due to an external tool failing. To debug the issue:

  1. If you are testing on a 64-bit system, ensure it can compile 32-bit binaries. If necessary, follow the instructions in Unit tests aren’t running on my 64-bit machine to enable cross-compilation.
  2. If problems persist, consult the documentation for the external tools to debug the issue.

Unit tests aren’t running on my 64-bit machine

SEL builds 32-bit binaries for most unit tests.

If you are running tests on a 64-bit machine, you must enable cross-compilation by executing 

sudo apt-get install gcc-multilib g++-multilib
